vibe codingで作ったアプリを投稿・発見できるギャラリーサイト「[Vibe Gallery](https://vibe-gallery.space)」をつくった。

## なぜつくったか

vibe codingで作ったアプリが、できあがってもそのまま放置になってしまうことが多かった。
置き場所として作った。

タグやAIツールで絞り込めるので、誰がどのツールでどんなものを作っているのかを眺める用途にも使える。

## できること
- アプリの投稿（タイトル / 説明 / サムネイル / websiteUrl / GitHub URL / タグ / 使ったAIツール）
- タグやAIツールでの絞り込み
- アプリごとの「ビハインドシーン」記事の投稿（開発の裏話・工夫したところを後から書き足せる）
- サイト所有確認バッジ
- MCPサーバ経由でClaudeなどから直接投稿・更新

## コアにしたこと

今回サイトを作る上で、特に時間をかけて考えたのは次の3つ。

### 1. サイト所有確認

Webサイトという特性上、URLさえ知っていれば誰でも投稿できてしまう。
そこで、登録するwebsiteUrlに対して**そのサイトの作者であること**の確認を入れるようにした。

仕組みはシンプルで、各ユーザーには Vibe Gallery 上のprofile IDが振られていて、そのIDを次のいずれかの形でサイトのルートHTMLに埋め込んでもらう。

```html
<!-- 方法1: meta タグ -->
<meta name="vibe-gallery-verification" content="{あなたのprofile ID}">

<!-- 方法2: html の data 属性 -->
<html lang="ja" data-vibe-gallery-verification="{あなたのprofile ID}">
```

サーバ側は登録されたURLの`origin /` を取得しに行き、`<html>`タグの`data-vibe-gallery-verification`属性、または`<head>`内の`<meta name="vibe-gallery-verification">`のどちらか一方でも一致すればverified扱いにする。
現在はrootだけを見ている。

所有確認ができるとアプリにバッジが付く。

### 2. MCPからの登録

このサイトの一番気に入っているところで、Claude Desktopなどから話しかけるだけでアプリの投稿・更新・削除・ビハインドシーン記事の作成までできる。
WebUIにアクセスせずに、コードを書いている流れのままClaudeに「これさっきデプロイしたやつ、Vibe Galleryにも上げといて」と言うだけで投稿される。

実装は[Vercelの`mcp-handler`](https://vercel.com/changelog/mcp-server-support-on-vercel)とClerkの[`@clerk/mcp-tools`](https://github.com/clerk/mcp-tools)の組み合わせで、Streamable HTTP transportで`/api/mcp`に生やしている。

認証はClerk OAuth 2.1（PKCE + Dynamic Client Registration）。
読み書き問わず全ツールを`withMcpAuth({ required: true })`で守っているので、トークンなしで叩くと401と`WWW-Authenticate: Bearer resource_metadata="…"`が返り、それをトリガーにMCPクライアント側がOAuthフロー（DCR + ブラウザサインイン）を起動する流れ。

Claude Code / Claude Desktop / Cursor / VS Code それぞれの設定の仕方は[`/docs/mcp/install`](https://vibe-gallery.space/docs/mcp/install)に置いてある。
たとえばClaude Codeなら、`mcp-remote`を経由せずに次のコマンド一発で追加できる。

```bash
claude mcp add --transport http vibe-gallery https://vibe-gallery.space/api/mcp
```

初回接続時にブラウザが開いてClerkのサインインを通れば、以降はClaudeから`submit_app`が呼べる。

提供しているツールは大体こんな構成。

| 種別 | tool | 役割 |
| --- | --- | --- |
| 読み取り | `list_apps` / `get_app` / `list_user_apps` / `list_ai_tools` | ギャラリーを眺める |
| 読み取り | `list_behind_scenes` / `get_behind_scenes` | 開発記事を読む |
| 書き込み | `get_me` | 認証中ユーザーのprofile |
| 書き込み | `get_verification_info` | 所有確認用のスニペットを取得 |
| 書き込み | `create_upload_url` / `open_thumbnail_uploader` | サムネイルのアップロード |
| 書き込み | `submit_app` / `update_app` / `delete_app` / `reverify_app` | アプリ投稿の一通り |
| 書き込み | `create_behind_scenes` / `update_behind_scenes` / `delete_behind_scenes` | 開発記事の一通り |

`submit_app`は`websiteUrl`が指定されていれば内部で所有確認まで一気に走らせるので、ユーザー側は「このアプリ投稿しといて」と言うだけで、所有確認 → 投稿 → 検証バッジ付与までが完結する。
所有確認用のIDが分からなければ`get_verification_info`を呼べば、そのままmetaタグ・data属性のスニペットまで返してくれるので、ClaudeにHTMLに埋め込んでもらってから投稿、という流れが自然に取れる。

サムネイルは署名付きURLによるダイレクトアップロードにしている。

[MCP Apps仕様（2026-01-26）](https://modelcontextprotocol.io/)を使った`open_thumbnail_uploader`のインタラクティブな経路もあるにはあるが、まだうまく動かないので引き続き実装していく。

### 3. ビハインドシーン

アプリ投稿だけだと「投稿した瞬間がピーク」になりがちで、作った直後はテンションが高いので説明を書くが、その後の改善や試行錯誤は書く場所がなくて消えていく。

そこでアプリごとに**ビハインドシーン記事**をぶら下げられるようにした。
1アプリに対してN本の記事をMarkdownで書ける、ミニブログみたいなもの。

例えば「[Claude DesignとClaude Codeで作り直した](https://vibe-gallery.space/apps/7c95c821-dff9-4cdf-9603-c79def95b2f2/behind-scenes/7433bde9-c2c2-4968-a449-e678fc83f7d0)」のような、メインの作品ページとは分けたい粒度の話を書ける場所として運用している。

ビハインドシーン記事もMCPから書ける（`create_behind_scenes` / `update_behind_scenes`）ので、Claude Codeで実装した直後に「今日やったこと、ビハインドシーンに書いといて」と言えばそのままドラフトに入る。
記事は`status: "draft" | "published"`で公開状態を切り替えられて、デフォルトは下書き。意図せず公開されない安全側に倒している。

## 技術スタック

- **Framework**: Next.js 14 App Router (RSC)
- **DB**: Postgres + Drizzle ORM
- **認証**: Clerk（OAuth 2.1 / DCR込み）
- **MCP**: `mcp-handler` + `@clerk/mcp-tools`
- **Storage**: S3互換（ローカルはrustfs）
- **UI**: Tailwind + shadcn/ui
- **Deploy**: Vercel

mutationはServer Actionsを使わず、`app/api/*/route.ts`に切る方針にしている。
WebからもMCPからも同じrepository関数を叩くようにしておきたかったので、ロジックを`shared/repositories/*`に寄せて両方から呼ぶ形にした。
おかげでMCPツールは「認証 → 入力検証 → repository関数を呼ぶ」だけの薄いラッパで済んでいる。

## 感想
