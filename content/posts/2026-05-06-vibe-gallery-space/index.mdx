![Vibe Gallery](./cover.png)

vibe codingで作ったアプリを投稿・発見できるギャラリーサイト「[Vibe Gallery](https://vibe-gallery.space)」をつくった。

## できるようになること

- 自分が作ったアプリを投稿・公開できる（タイトル / 説明 / サムネ / websiteUrl / GitHub URL / タグ / 使ったAIツール）
- タグやAIツールで他の人の作品を探せる
- アプリごとに「ビハインドシーン」記事として、開発過程や考えを書き残せる
- 自分のサイトであることを確認した上で所有確認バッジを付けられる
- ClaudeなどのMCPクライアントから直接投稿・更新できる

## なぜつくったか

vibe codingしたものを公開する場が欲しかった。
ZennやQiitaは技術ナレッジを書く場所というイメージで、単なるアプリ投稿の置き場とはちょっと違うなと個人的に思っていた。
作ったものそのものを投稿しつつ、作っている過程の記録も同じ場所で書けるサービスがほしくて、自分で作った。

## コアにしたこと

作るうえで時間をかけたのは次の3つ。

### 1. サイト所有確認

Webサイトという特性上、URLさえ知っていれば誰でも投稿できてしまう。そこで、登録するURLに対して**そのサイトの作者であること**の確認を入れた。

ユーザーごとに振られたprofile IDを、サイトのルートHTMLに次のいずれかの形で埋め込んでもらう。

```html
<!-- 方法1: meta タグ -->
<meta name="vibe-gallery-verification" content="{あなたのprofile ID}">

<!-- 方法2: html の data 属性 -->
<html lang="ja" data-vibe-gallery-verification="{あなたのprofile ID}">
```

2方式あるのは、プラットフォームによっては`<meta>`タグを自由に追加できないことがあるため。その場合は`<html>`要素のdata属性のほうを使う。

サーバ側はoriginにGETを投げて、`data-vibe-gallery-verification`属性か`<meta name="vibe-gallery-verification">`のどちらかでprofile IDが見つかれば確認OK。現在はrootだけを見ている。

確認が通ったアプリにはバッジが付く。たとえばこのブログ自身もVibe Galleryに登録してあって、yukyu.netの横に🔵のバッジが付いている。確認が通っていないアプリも普通に表示される。

![Vibe Galleryにおけるyukyu.netのアプリページ。所有確認バッジが付いている](./vibe-gallery-app-page.png)

### 2. MCPからの登録

Claude Desktopなどから話しかけるだけで、アプリの投稿・更新・削除、ビハインドシーン記事の作成までできるようにしている。
Webを開かずに、コードを書いている流れのまま「これさっきデプロイしたやつ、Vibe Galleryにも上げといて」と言うだけで投稿が完了する。

実装は[Vercelの`mcp-handler`](https://vercel.com/changelog/mcp-server-support-on-vercel)とClerkの[`@clerk/mcp-tools`](https://github.com/clerk/mcp-tools)の組み合わせで、Streamable HTTP transportで`/api/mcp`に生やしている。

認証はClerk OAuth 2.1（PKCE + Dynamic Client Registration）。読み書き問わず全ツールを`withMcpAuth({ required: true })`で保護していて、トークンなしで叩くと401と`WWW-Authenticate: Bearer`を返す。これをきっかけにMCPクライアント側でOAuthフロー（DCR + ブラウザサインイン）が走る。

Claude Code / Claude Desktop / Cursor / VS Code 向けの設定は[`/docs/mcp/install`](https://vibe-gallery.space/docs/mcp/install)にまとめてある。たとえばClaude Codeなら次のコマンド一発で追加できる。

```bash
claude mcp add --transport http vibe-gallery https://vibe-gallery.space/api/mcp
```

初回接続時にブラウザでClerkにサインインすれば、以降はClaudeから`submit_app`等が呼べる。

提供しているツールはこんな構成。

| 種別 | tool | 役割 |
| --- | --- | --- |
| 読み取り | `list_apps` / `get_app` / `list_user_apps` / `list_ai_tools` | ギャラリーを眺める |
| 読み取り | `list_behind_scenes` / `get_behind_scenes` | 開発記事を読む |
| 書き込み | `get_me` | 認証中ユーザーのprofile |
| 書き込み | `get_verification_info` | 所有確認用のスニペットを取得 |
| 書き込み | `create_upload_url` / `open_thumbnail_uploader` | サムネイルのアップロード |
| 書き込み | `submit_app` / `update_app` / `delete_app` / `reverify_app` | アプリ投稿の一通り |
| 書き込み | `create_behind_scenes` / `update_behind_scenes` / `delete_behind_scenes` | 開発記事の一通り |

`submit_app`は`websiteUrl`があれば内部で所有確認も一緒に走らせる。「このアプリ投稿しといて」と言うだけで、所有確認 → 投稿 → バッジ付与までが完結する。
所有確認用のIDがなくても、`get_verification_info`を呼べばmetaタグ・data属性のスニペットを返してくれる。ClaudeにそれをHTMLに埋め込んでもらってから投稿、という流れも自然に取れる。

サムネイルは署名付きURLによるダイレクトアップロード。
[MCP Apps仕様（2026-01-26）](https://modelcontextprotocol.io/)を使った`open_thumbnail_uploader`によるインタラクティブな経路も用意してあるが、まだうまく動かないので引き続き実装していく。

### 3. ビハインドシーン

アプリページとは別に、開発の過程や試行錯誤、後からの改善などを書き残せる**ビハインドシーン記事**をアプリごとにぶら下げられるようにした。1アプリに何本でもMarkdownで書ける、ミニブログみたいなもの。

「メイキング」だと作っている過程に寄りすぎる気がして、信念や考え方も同じ場所に書けるように「ビハインドシーン」という名前にした。

例えば「[ブログを Claude Design × Claude Code で作り直した](https://vibe-gallery.space/apps/7c95c821-dff9-4cdf-9603-c79def95b2f2/behind-scenes/7433bde9-c2c2-4968-a449-e678fc83f7d0)」のような、メインの作品ページとは分けたい粒度の話を書く場所として運用している。

記事もMCPから書ける（`create_behind_scenes` / `update_behind_scenes`）ので、Claude Codeで実装した直後に「今日やったこと、ビハインドシーンに書いといて」と言えばそのまま下書きに入る。デフォルトはdraftなので、意図せず公開されることはない。

## llms.txt

公開されているすべてのページに`llms.txt`を持たせている。投稿されたアプリやビハインドシーン記事も含めて、URLの末尾に`/llms.txt`を付ければそのページのLLM向けMarkdownが返ってくる。

## 技術スタック

- **Framework**: Next.js 14 App Router (RSC)
- **DB**: Postgres + Drizzle ORM
- **認証**: Clerk（OAuth 2.1 / DCR込み）
- **MCP**: `mcp-handler` + `@clerk/mcp-tools`
- **Storage**: S3互換（ローカルはrustfs）
- **UI**: Tailwind + shadcn/ui
- **Deploy**: Vercel

mutationはServer Actionsを使わず`app/api/*/route.ts`に切っている。WebもMCPも同じrepository関数を叩く構成にしたので、MCPツールは「認証 → 入力検証 → repository関数を呼ぶ」だけの薄いラッパで済んでいる。
